# 尝试给 Skill 写测试

我有几个 skill 处理前端 bug：用户报一个问题，skill 先把它归类（白屏 / 点击无反应 / 图表错 / …），再路由到对应的下游 skill 去修。这套"分类 + 路由"是它的核心决策。问题是：skill 本质是 prompt，我改一版，怎么确定它的分类和路由没退化？LLM 输出不确定，没法像普通代码那样直接断言。

## 不能测文字

第一版我断言 skill 的回复文本——比如"回复里应包含‘运行时’"。两个毛病：

- **脆**：模型换个措辞（"运行态""页面活态"）就匹配失败，可决策是对的。我验的是措辞，不是决策。
- **flaky（时过时不过）**：同一条用例，跑两次结果不一样。

根因：回复文本不是决策的可靠投影。要测决策，就得让决策本身变成一个**确定的、可观测的信号**。

## 让 skill 把决策显式吐出来

做法：在分类/路由这个决策点，让 skill 调一个固定工具，把决策写成结构化的一行：

```bash
skill-trace '{"checkpoint":"intake","scenario":"crash","route":"trantor-fe-runtime"}'
```

`scenario` 是它判定的类别，`route` 是它要转发的下游 skill。SKILL.md 里把这步定成硬约束："分类后、回复前必须发一次；漏发 = 不可观测 = 失败"。

关键：这是个**工具调用，不是回复文本**——生产环境只把 skill 给用户的那段 text 转出去（比如钉钉），用户根本看不到这条 trace。所以它是一条干净的测试旁路，不污染真实回复。

## 测试怎么读到它

测试 runner 用 `claude --output-format stream-json` 跑这个 skill，从输出流里解析出每一次工具调用；遇到 `skill-trace`，把那行 JSON 提出来单独归到 checkpoints（和普通工具调用分开），其余工具按时序记下。这样一次运行就被还原成一条结构化 trace：决策 checkpoint + 工具序列。

## 断言只认决策符号

断言写成一小段 Clojure s-expr（整个 harness 跑在 Babashka 上，s-expr 是原生的；解释器自己写，只认固定几个谓词，可复用）：

```clojure
(checkpoint :scenario "crash" :route "trantor-fe-runtime")  ; 决策字段精确匹配
(checkpoint-before-explore :intake)                          ; 纪律：先决策，别没想清就翻代码
```

测的是符号，不是文字。文字能改写，这条决策记录改不了。

## 让它确定、能看趋势

- **跑在假数据上**：skill 里调下游服务的地方指向一个 mock 网关 + fixture（schema 树、workspace 假数据），不连真后端、不开真浏览器——每次结果一致，可反复比。
- **健康曲线**：每条用例每次的通过率写进 `scores.jsonl`。单次绿不算数，跑 N 次看通过率；flaky 和回归都在曲线上现形。

## 反向验证

最后一步：手动把某条路由改错，跑测试，看分数掉没掉到 0。掉了，才证明这套断言真能咬住错误——否则你不知道它会不会永远绿。这是给"测试本身"做的测试。
